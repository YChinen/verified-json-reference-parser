# Roadmap（開発計画）

本ドキュメントは `verified-json-reference-parser` の開発方針と段階的目標を示します。

本プロジェクトは、理論的完全性よりも **実用的に十分な保証** を優先します。

---

## 基本方針

* 意味論コアは小さく鋭く保つ
* 重要な不変条件を形式的に記述する
* SMT による差分検証で実装の健全性を担保する
* JSON Schema 全体の実装には踏み込まない
* 意味論が確定するまでは拡張より安定を優先する

---

# Phase 1 — Verified JSON Pointer コア

**目標:** JSON Pointer の意味論を形式的に固定する。

## スコープ

* RFC 6901 に基づく解析（parse）
* エスケープ / アンエスケープ
* ポインタの文字列化（format）
* 決定的なポインタ評価（`get`）
* ローカル JSON Reference（`#`, `#/...`）

## 検証

### Rocq

* `unescape(escape(x)) = x`
* `parse(format(ptr)) = ptr`
* ポインタ評価の決定性

### Z3

* 反例生成
* TypeScript 実装との差分検証
* Phase 1 の具体ケースに対する Z3 ベースのオラクル
* 回帰テストケースの自動蓄積

## 完了条件

* コア API が安定する
* 差分検証が CI に組み込まれる
* 形式的に固定された不変条件が明文化される

---

# Phase 2 — 検証基盤の強化

**目標:** 既存の Z3 ベース検証フローを強化し、運用しやすくする。

## スコープ

* Z3 オラクルの性能改善と堅牢化
* 反例の最小化
* 差分検証の自動化強化
* Python baseline と Z3 オラクルの parity 検証強化

## 明確な非目標

* RFC 3986 の完全検証
* 外部参照の取得
* JSON Schema キーワード評価

## 完了条件

* Z3 オラクルと TS 実装の整合性が継続的に保たれる
* 発見された反例が自動的に回帰テストに追加される
* 意味論の境界が明確に維持される

---

# Phase 3 — URI 対応（必要部分のみ）

**目標:** 実用上必要な範囲で URI 解決を導入する。

## スコープ

* Base URI 解決
* `$id` の基本対応
* RFC 3986 の必要部分のみ

## 非目標

* RFC 3986 の完全形式検証
* URI 正規化の厳密証明

---

# Phase 4 — 意味論の凍結と安定化

**目標:** 意味論コアを固定し、API を安定化させる。

## タスク

* コア不変条件の最終確認
* 公開 API の固定
* セマンティックレイヤーの安定宣言
* 内部実装変更の影響最小化

---

# 長期ビジョン

本プロジェクトは以下を目指す。

* 検証指向ライブラリ設計の実践例
* JSON Schema エンジンの基盤部品
* 形式的システム開発のケーススタディ

意味論コアは、形式的に固定された時点で安定化することを目標とする。
