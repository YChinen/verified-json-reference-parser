# CONTRIBUTING（貢献ガイドライン）

`verified-json-reference-parser` に関心を持っていただきありがとうございます。

本プロジェクトは、機能拡張よりも **意味論の安定性と実用的保証** を優先します。

プルリクエストを送る前に、以下の方針をご確認ください。

---

## 1. スコープと境界

本ライブラリは以下に限定されます。

* JSON Pointer（RFC 6901）
* ローカル JSON Reference（`#`, `#/...`）

以下は対象外です。

* JSON Schema キーワード評価
* 外部参照の取得
* RFC 3986 の完全実装
* ネットワーク関連機能

スコープを拡張する提案は、原則として慎重に扱われます。

---

## 2. 意味論の安定性を最優先

本プロジェクトの中核は「意味論コア」です。

以下の変更は避けてください。

* 振る舞いを暗黙に変更する修正
* 暗黙的な補正や正規化の追加
* 説明なく意味論を拡張する変更

振る舞いを変更する場合は、

* 変更点を明確に文書化する
* 回帰テストを追加する
* 差分検証を通過させる

ことが必要です。

---

## 3. 検証要件

振る舞いに影響する変更は、以下を満たす必要があります。

* 単体テスト
* ゴールデンテスト（該当する場合）
* Z3 オラクルとの差分検証

意味論に変更がある場合は、

* Rocq の形式仕様
* Z3 モデル

の更新が必要になる可能性があります。

---

## 4. 公開 API ポリシー

公開 API は最小限に保たれています。

* 不必要なエクスポートは追加しない
* 内部実装は公開しない
* 破壊的変更は事前に議論する

安定性を重視します。

---

## 5. 実装方針

* コアロジックは純粋関数として実装する
* 副作用を持ち込まない
* 実行時依存を増やさない
* 明確さを優先する

複雑さよりも透明性を重視します。

---

## 6. Verification 層について

`verification/` ディレクトリは開発専用です。

* npm パッケージには含まれません
* 実行時依存にはなりません
* Z3 は限定オラクルとして使用されます

意味論エンジンの拡張は、明確な境界の範囲内でのみ検討されます。

---

## 7. 設計思想

本プロジェクトは以下を重視します。

* 小さく明確な責務
* 明示的な不変条件
* 実用的保証
* 長期的な安定性

機能追加よりも意味論の明確性を優先します。

---

## 8. 事前相談の推奨

大きな変更や意味論に影響する提案については、
事前に Issue で相談してください。

疑わしい場合は、

> 機能追加よりも意味論の明確性を優先する

という原則を採用します。

---
