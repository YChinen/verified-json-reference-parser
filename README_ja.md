# verified-json-reference-parser

> JSON Schema エンジンのための、実用的保証付き JSON Reference コア実装

---

## 概要

`verified-json-reference-parser` は、JSON Schema 実装に必要となる以下の基盤部分を対象とした、形式仕様付きコアライブラリです。

* JSON Pointer（RFC 6901）
* ローカル JSON Reference（`$ref`）
* 参照解決に必要な基礎構造

本プロジェクトは「完全な形式証明」を目指すものではありません。
**実用的に十分な保証を提供すること**を目的としています。

---

## 背景

JSON Schema の実装では、以下の処理が中核になります。

* JSON Pointer の解析・評価
* `$ref` の fragment 解決
* URI を含む参照処理

これらは仕様上は小さな要素ですが、

* `$ref` 解決の不整合
* バリデータ間の挙動差異
* エラーパスの不一致
* デバッグ困難な参照バグ

といった問題の温床になりやすい領域です。

既存実装の多くは実用上は十分に動作しますが、
形式的な意味論が明示されているわけではありません。

本プロジェクトは、**検証に基づいた参照解決のコア**を提供することを目指します。

---

## 本プロジェクトにおける「Verified」の意味

本ライブラリは、すべてを形式的に証明することを主張しません。

ここでいう「Verified」とは、以下を意味します。

* コア意味論が形式的に記述されている
* 重要な不変条件が機械的に検証されている
* TypeScript 実装が検証用オラクルと参照モデルに対して継続的に照合されている
* Z3 ベースのオラクルにより、Phase 1 の具体ケースに対する差分検証を行っている
* SMT ソルバによる反例探索を用いて意味論差分を探索している
* 発見された反例は回帰コーパスに昇格できる

理論的完全性よりも、実運用で信頼できる保証を優先します。

---

## 現在のスコープ（Phase 1）

### RFC 6901 — JSON Pointer

* ポインタ文字列の解析（parse）
* ポインタの文字列化（format）
* トークンのエスケープ / アンエスケープ
* 決定的なポインタ評価（`get`）

設計方針：

* 内部表現は `readonly string[]`（トークン配列）
* 不正なエスケープはエラーとして扱う
* 評価は決定的である（同じ入力は常に同じ結果を返す）
* RFC 要件を超える暗黙的な補正は行わない

---

### ローカル JSON Reference

* `#`
* `#/...`
* fragment の分離
* JSON Pointer に基づくドキュメント内解決

外部ドキュメントの取得は現段階では対象外とします。

---

## 検証戦略

### Rocq（Coq）

以下の性質を形式的に定義し、機械的に検証します。

* `unescape(escape(x)) = x`
* `parse(format(ptr)) = ptr`
* ポインタ評価の決定性

意味論の中核を固定する役割を担います。

---

### Z3（SMT による検証）

* `get` / `resolveLocalRef` の具体ケースに対する full-result オラクル評価
* 反例生成
* TypeScript 実装との差分検証
* 回帰テストケースの自動増殖

検証ツール群では、純粋な Python 参照モデルと Z3 ベースのオラクルを併用します。
Z3 は完全証明器ではなく、意味論オラクルおよび差分検出エンジンとして利用します。

---

## ロードマップ

### Phase 2

* RFC 3986 の必要部分に基づく URI 解決
* Base URI の扱い
* `$id` 対応

### Phase 3

* JSON Reference の拡張対応
* 外部参照解決フック
* より高度な参照解決戦略

---

## 非目標（現時点）

* RFC 3986 の完全な形式検証
* ネットワーク取得機能
* JSON Schema 全体の検証ロジック
* エラーメッセージ仕様の厳密化

本ライブラリは **参照解決の正確性** に特化します。

---

## 想定用途

* JSON Schema バリデータの基盤部品
* 検証付きリファレンス実装
* 検証指向ライブラリ設計のケーススタディ

---

## 設計思想

* 理論的完全性よりも実用的保証を優先
* 小さく明確な意味論コア
* 明示的な不変条件
* 継続的な差分検証
* 意味論が凍結された後は安定性を重視

コア意味論は、形式的に固定された時点で安定化することを目標としています。
